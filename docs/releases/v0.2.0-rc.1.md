# v0.2.0-rc.1 service catalog and recovery

This release candidate expands Agent Homelab from ingress operations into application bring-up and tested recovery planning.

## Included

- a sanitized catalog for twelve service patterns observed in a multi-stack edge-gateway homelab
- official-source, ingress, canonical-URL, persistence, backup-consistency, and gotcha metadata for every recipe
- installed `agent-homelab recipe list/show` commands
- a BorgBackup 1.4 runner with per-stack encrypted repositories
- Compose, source-path, named-volume, and repository preflight before downtime
- named Docker volume export, stopped-stack archive creation, retention pruning, compaction, and guaranteed restart attempt
- repository listing and integrity checks plus non-overlapping staging restores
- daily systemd service/timer examples
- `homelab-service-catalog` and `homelab-borg-backup` agent skills
- first-homelab, service-catalog, Borg, and expanded disaster-recovery documentation

## Safety boundaries

- Real service configuration, domains, paths, repository locations, passphrases, keys, and restore output remain ignored runtime state.
- The Borg runner rejects inline secret fields and repositories nested beneath backup sources.
- Restore never writes directly into configured live paths or Docker volumes.
- Application-native backup requirements remain authoritative for services such as Nextcloud AIO, GitLab, Odoo, and WordPress.
- Complex application Compose files remain sourced from their upstream maintainers rather than copied from a private deployment.

## Compatibility

- Python 3.11 or newer
- Docker Compose v2
- BorgBackup 1.4.x
- BusyBox 1.37.0 by default for named-volume export

This release does not automate Portainer API lifecycle control, remote filesystem backup over SSH, live-path restore, Borg repair, or named-volume rehydration. Those remain explicit operator actions.
